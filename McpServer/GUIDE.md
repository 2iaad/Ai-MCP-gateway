## 1. What is MCP?

**MCP stands for Model Context Protocol.**

Think of it like a USB standard for AI.

Before MCP, every AI tool had its own custom way of connecting to external data (files, databases, APIs). MCP is a *single open standard* that solves this once and for all.

```
┌───────────────────────────────────────────────────────────────┐
│                        AI Assistant                           │
│                   (e.g. GitHub Copilot)                       │
└──────────────────────────┬────────────────────────────────────┘
                           │  speaks MCP
           ┌───────────────▼───────────────┐
           │          MCP Server           │
           │    (the code we just built)   │
           └──────┬──────────────┬─────────┘
                  │              │
          ┌───────▼──┐    ┌──────▼──────┐
          │  Your    │    │  Context7   │
          │  Files   │    │    API      │
          └──────────┘    └─────────────┘
```

The AI talks to YOUR server using MCP. YOUR server talks to the real world (files, APIs). This means the AI never needs to know the details of how files are read or how Context7 works — it just calls the tools you expose.

## 2. What is Context7?

**Context7** is a service that provides **up-to-date, version-accurate documentation** for popular libraries (React, Next.js, TypeScript, etc.).

The problem it solves: AI models are trained on data with a cutoff date. If you ask Copilot about the latest version of a library, it might give you outdated answers from its training data.

Context7 fixes this by serving live docs that the AI can fetch *at the moment you ask*.

The tool in our server (`get_library_docs`) calls two Context7 API endpoints:

| Step | Endpoint | Purpose |
|------|----------|---------|
| 1 | `GET /api/v1/search?query=react` | Find the internal ID for the library |
| 2 | `GET /api/v1/{library-id}?topic=hooks&tokens=8000` | Get the actual documentation |

## 4. Step-by-Step: What Was Built

### Step 1 – Scaffolded the project (`package.json` + `tsconfig.json`)

`package.json` is the configuration file every Node.js project has. It lists:
- The project name and version
- Scripts (`build`, `start`, `dev`)
- Dependencies (packages we need at runtime)
- DevDependencies (packages we only need during development)

**Key dependencies:**

| Package | Why we need it |
|---------|---------------|
| `@modelcontextprotocol/sdk` | Gives us the `Server` class and all MCP plumbing |
| `typescript` | Lets us write typed TypeScript instead of plain JavaScript |
| `tsx` | Runs `.ts` files directly without a separate build step (great for development) |
| `@types/node` | TypeScript type definitions for Node.js built-ins like `fs` and `path` |

`tsconfig.json` tells the TypeScript compiler how to turn our `.ts` files into `.js` files:
- `"module": "NodeNext"` – use modern ES module syntax (`import`/`export`)
- `"outDir": "./dist"` – put compiled files into the `dist/` folder
- `"strict": true` – catch more potential bugs at compile time

### Step 2 – Installed dependencies

```bash
npm install
```

This reads `package.json` and downloads all listed packages into `node_modules/`.

### Step 3 – Wrote the server (`src/index.ts`)

The server has 4 parts (see the file for detailed comments):

1. **Create the server** – `new Server(...)` initialises the MCP server with its name and capabilities
2. **Advertise the tools** – `ListToolsRequestSchema` handler returns the list of tools with their descriptions and input schemas
3. **Handle tool calls** – `CallToolRequestSchema` handler runs the actual logic when the AI calls a tool
4. **Start the server** – `StdioServerTransport` wires stdin/stdout and `server.connect()` starts listening

### Step 4 – Compiled TypeScript to JavaScript

```bash
npm run build
```

The TypeScript compiler (`tsc`) reads `src/index.ts` and outputs `dist/index.js`. VS Code launches the compiled `.js` file (Node.js can't run `.ts` directly).

### Step 5 – Created `.vscode/mcp.json`

This file is how VS Code discovers your MCP server. When VS Code starts, it reads this file and launches the server process. Copilot Chat in **Agent mode** then sees your tools in its tool list.

## 5. How the Tools Work

### Tool 1: `read_file`

```
AI calls: read_file({ file_path: "/Users/ziyad/Desktop/Prompt-engineering/space/context.md" })
              ↓
Server runs: fs.readFile(file_path, "utf-8")
              ↓
Returns: the file content as plain text
```

Use cases:
- Load a context file to give Copilot background information
- Read a skill document before asking it to follow specific guidelines

### Tool 2: `list_directory`

```
AI calls: list_directory({ dir_path: "/Users/ziyad/Desktop/Prompt-engineering/space" })
              ↓
Server runs: fs.readdir(dir_path, { withFileTypes: true })
              ↓
Returns: a list of files and folders with [FILE] / [DIR] labels
```

Use cases:
- Discover which context/skill files exist before reading them

### Tool 3: `get_library_docs`

```
AI calls: get_library_docs({ library_name: "nextjs", topic: "routing" })
              ↓
Step A – Search: GET https://context7.com/api/v1/search?query=nextjs
              ↓  gets back library ID, e.g. "/vercel/next.js"
Step B – Fetch:  GET https://context7.com/api/v1/vercel/next.js?topic=routing&tokens=8000
              ↓
Returns: up-to-date Markdown documentation
```

## 6. How VS Code Connects to the Server

The file `.vscode/mcp.json` contains:

```json
{
  "servers": {
    "my-mcp-server": {
      "type": "stdio",
      "command": "node",
      "args": ["${workspaceFolder}/mcp-server/dist/index.js"]
    }
  }
}
```

**What each field means:**

| Field | Meaning |
|-------|---------|
| `"type": "stdio"` | VS Code will start the server as a subprocess and talk to it via standard input/output |
| `"command": "node"` | The executable to run |
| `"args"` | Arguments passed to node – the path to our compiled server |
| `${workspaceFolder}` | A VS Code variable that resolves to the root of your workspace |

**The flow:**

```
VS Code opens workspace
      ↓
Reads .vscode/mcp.json
      ↓
Spawns: node /path/to/mcp-server/dist/index.js
      ↓
Our server starts, writes "MCP server started" to stderr
      ↓
Copilot Agent mode can now see:
  • read_file
  • list_directory
  • get_library_docs
```

## 7. How to Run & Use It

### Prerequisites

- Node.js 18 or later (`node --version`)
- VS Code with the **GitHub Copilot** extension installed

### First-time setup (already done)

```bash
cd mcp-server
npm install       # install dependencies
npm run build     # compile TypeScript → JavaScript
```

> After any change to `src/index.ts`, run `npm run build` again.

### Using the tools in Copilot Chat

1. Open **Copilot Chat** in VS Code (`Ctrl+Alt+I` / `Cmd+Alt+I`)
2. Switch to **Agent mode** (click the mode selector at the top of the chat panel and choose **Agent**)
3. You should see a tools icon (🔧) — your tools will appear there
4. Now ask questions that use the tools, for example:

```
Read the file /Users/ziyad/Desktop/Prompt-engineering/space/context.md
and use it as context to answer my question.
```

```
Fetch the latest React documentation on hooks and explain
the difference between useEffect and useLayoutEffect.
```

```
List the files in /Users/ziyad/Desktop/Prompt-engineering/space
and read each one to understand the project context.
```

Copilot will automatically call the appropriate tool without you needing to trigger it manually.

## 8. How to Add More Tools

To add a new tool, you only need to touch **two places** in `src/index.ts`:

### A – Add the tool definition in the `ListToolsRequestSchema` handler

```typescript
{
  name: "my_new_tool",
  description: "What this tool does and when the AI should use it.",
  inputSchema: {
    type: "object",
    properties: {
      my_argument: {
        type: "string",
        description: "What this argument is for",
      },
    },
    required: ["my_argument"],
  },
},
```

### B – Add the handler in the `CallToolRequestSchema` handler

```typescript
if (name === "my_new_tool") {
  const { my_argument } = args as { my_argument: string };

  // your logic here

  return {
    content: [{ type: "text", text: "your result here" }],
  };
}
```

Then rebuild:

```bash
npm run build
```

And restart VS Code (or reload the MCP server) to pick up the new tool.

## 9. Troubleshooting

### "Server not found" or tools don't appear

- Make sure you ran `npm run build` and that `dist/index.js` exists
- Reload VS Code: `Cmd+Shift+P` → `Developer: Reload Window`
- Check Output panel: `Cmd+Shift+U` → select "MCP" from the dropdown

### "Error reading file"

- The path must be **absolute** (starts with `/`)
- Check that the file actually exists and you have read permissions
- Example of a correct path: `/Users/ziyad/Desktop/Prompt-engineering/space/context.md`

### "No documentation found for X on Context7"

- Try different spellings: `nextjs` vs `next.js`, `react` vs `reactjs`
- Check your internet connection (the tool makes live HTTP requests)

### TypeScript compilation errors

```bash
cd mcp-server && npm run build
```

Read the error message – it will tell you exactly which file and line number has the problem.
